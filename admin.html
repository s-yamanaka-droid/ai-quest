🗄 データベース設計
Supabase PostgreSQL — テーブル構成とスキーマ
players テーブル
CREATE TABLE players (
  id          uuid PRIMARY KEY
              DEFAULT gen_random_uuid(),
  name        text NOT NULL,
  created_at  timestamptz DEFAULT now(),
  last_seen_at timestamptz DEFAULT now()
);
カラム型説明
iduuidプレイヤーID（PK）
nametextログイン名（まな 等）
created_attimestamptz初回登録日時
last_seen_attimestamptz最終アクセス日時
progress テーブル
CREATE TABLE progress (
  id         uuid PRIMARY KEY
             DEFAULT gen_random_uuid(),
  player_id  uuid REFERENCES players(id)
             ON DELETE CASCADE,
  lesson_lv  integer NOT NULL,
  xp_gained  integer NOT NULL DEFAULT 0,
  cleared_at timestamptz DEFAULT now(),
  UNIQUE(player_id, lesson_lv)
);
カラム型説明
player_iduuid FKplayers.id への外部キー
lesson_lvintegerレッスン番号（1〜50）
xp_gainedinteger獲得XP
cleared_attimestamptzクリア日時
ER図
players
PK id : uuid
name : text
created_at : timestamp
last_seen_at : timestamp
1 ──── ∞
progress
PK id : uuid
FK player_id : uuid
lesson_lv : integer
xp_gained : integer
cleared_at : timestamp
UNIQUE制約：(player_id, lesson_lv) — 同一プレイヤーが同じレッスンを二重登録できない
よく使うクエリ
-- プレイヤー一覧と進捗数
SELECT p.name, COUNT(pr.id) as cleared, SUM(pr.xp_gained) as total_xp
FROM players p
LEFT JOIN progress pr ON p.id = pr.player_id
GROUP BY p.id, p.name
ORDER BY total_xp DESC;

-- 特定プレイヤーの進捗詳細
SELECT lesson_lv, xp_gained, cleared_at
FROM progress
WHERE player_id = '...'
ORDER BY lesson_lv;

-- STAGE別クリア率
SELECT
  CEIL(lesson_lv / 5.0) as stage,
  COUNT(*) as total_clears
FROM progress
GROUP BY stage
ORDER BY stage;

設計思想
WHY RPG?
学習完了率を高めるため。通常の研修資料の完了率は30〜40%だが、ゲーミフィケーションで85%以上を目指す。XPとバッジが「続けたい」という動機を作る。
3ブロック構成
「読む→解く→やる」の順が最も記憶定着率が高い（ブルームのタキソノミー）。チャレンジをAllenに提出することでアウトプットを強制し、曖昧な理解を防ぐ。
順番固定の理由
STAGE 1〜5でAI基礎・実務スキルを固める。STAGE 6〜10で自動化・開発・案件獲得へと発展。前のSTAGEが基礎になるため、スキップ不可の設計にしている。
